4 Phase 0 — Prerequisites & Extraction

🎯 Purpose

Before any analysis begins, the engagement model must be determined and prerequisites satisfied. For Model A/B, this means establishing system connectivity. For Model C, this means the customer extracts 9 critical artifacts from their ECC system.

Model A C16 Connected to ECC — Setup Steps
  • Customer provides RFC connection details (host, system number, client, service port)
  • Customer creates a technical user with RFC + ADT access (roles: S_RFC, S_DEVELOP, read access to TADIR/TRDIR/DD03L/TFDIR)
  • Firewall rules opened for C16 → ECC connectivity (port 443 for ADT HTTPS or 33xx for RFC)
  • C16 validates connection: sap_test_connection + sap_list_programs
  • No extraction needed — C16 reads everything directly

Time: 1–3 days (mostly customer-side network/security approval)

Model B C16 Connected to S/4HANA — Setup Steps
  • Customer provides ADT HTTPS endpoint for S/4HANA (typically port 443 with path /sap/bc/adt/)
  • Customer creates a developer user with ADT + OData access
  • C16 validates connection and confirms S/4HANA release (≥ 2020)
  • C16 scans all custom Z/Y objects that were migrated or created
  • Customer provides ECC baseline documentation (if available) for comparison

Time: 1–3 days

8 Phase 2 — ASSESS (Fit-to-Standard & Gap Analysis)

🎯 Objective

For every business process and custom object identified in Phase 1, determine whether S/4HANA 2025 standard covers it — and document every gap with a closure strategy.

Fit-to-Standard Classification Framework

Every business requirement extracted from custom programs is classified into one of four categories:

FIT
S/4HANA standard covers this requirement exactly. No customization needed. Adopt the standard process.
FIT + CONFIG
Standard covers it, but requires configuration (IMG settings, BRF+ rules, workflow setup). No ABAP code needed.
FIT + EXTENSION
Standard covers ~70-80%. Remaining gap needs a Clean Core–compliant extension (released BAdI, Key User, or RAP side-by-side).
GAP
No standard equivalent exists. Must be justified, documented, and built as a Clean Core custom extension on S/4HANA or BTP.

9 Clean Core Classification Framework

Every custom object is classified against SAP's Clean Core principles — determining whether it uses only released, stable APIs or relies on internal/deprecated interfaces that will break in future upgrades.

Level A Cloud-Ready
Uses ONLY released APIs (CDS views, RAP BOs, released BAPIs with C1 contract). Deployable to S/4HANA Cloud. Future-proof.
Level B Classic Stable
Uses stable classic APIs (BAPIs, REUSE_ALV, CL_SALV_TABLE). Works on S/4HANA on-premise but NOT cloud-deployable.
Level C Non-Compliant
Direct SELECT on standard tables without CDS alternative, unreleased FMs, internal interfaces. Needs modernization.
Level D Critical
Deprecated patterns (SAPscript, removed tables), SAP standard modifications, EXEC SQL. Will break — mandatory rewrite.
📋 Classification Rules — How Objects Are Scored
Pattern DetectedLevelRationale
Uses only released APIs (C1 contract)ACloud-compatible
Uses CL_SALV_TABLE, REUSE_ALV_GRID_DISPLAYBStable on S/4HANA on-prem, not cloud-released
Uses stable BAPIs (BAPI_PO_CREATE1, etc.)BClassic APIs — work but not cloud-released
Direct SELECT on standard tables (LFA1, EKKO, VBAK)CCDS view alternatives exist — should modernize
Uses unreleased function modulesCMay change without notice in future releases

Overall Level Rule: The overall level of a program = MAX (worst) of all its dependencies. If a program uses one Level D pattern, the entire program is Level D.

📋 C16 Automation — How Classification Happens Per Model
Model A C16 → ECC

C16 reads every custom program's source code, resolves all dependencies (function modules, classes, tables, BAdIs), and classifies each dependency against SAP's cloudification catalog. The process:

  1. Fetch source via ADT
  2. Parse all external references (SELECT tables, CALL FUNCTION, class method calls, includes)
  3. Look up each reference in the cloudification catalog (Level A = released, Level B = classic, Level C = unreleased)
  4. Compute overall level = MAX(all dependencies)
  5. Generate per-program HTML report with score, findings, and remediation code

Speed: ~2 minutes per program. Batch mode: 50-100 programs per hour.

Model B C16 → S/4HANA

Same classification process, but on S/4HANA. Additional checks:

  1. Run ATC with ABAP_CLOUD_READINESS variant — SAP's own in-system classification
  2. Cross-reference with C16's cloudification catalog for accuracy
  3. Flag objects that passed migration but still use deprecated patterns (BSEG direct access, KONV, etc.)
  4. Identify "time bombs" — code that works today but will break on next S/4HANA upgrade
Model C Offline

Customer provides ABAP source files (.abap / .txt). C16 analyzes each file:

  1. Parse source code for table references, FM calls, class usage
  2. Classify against cloudification catalog (same rules)
  3. Generate reports — but without live system verification, some CDS view recommendations are marked "requires verification"

Also accepts SAP Readiness Check output for bulk classification.

10 The 5 Rs — Custom Code Disposition

Every custom object gets exactly ONE disposition. This determines its migration path.

🗑️
RETIRE
Criteria: Zero usage in ST03N, or redundant (another program does the same thing).
Action: Delete. Do not migrate. Document for audit trail.
🔄
REPLACE
Criteria: Standard S/4HANA Fiori app or transaction covers 90%+ of requirements.
Action: Adopt standard. Train users. Decommission custom.
🔧
REFIT
Criteria: Still needed but uses deprecated APIs or tables. Logic is valid, technology is not.
Action: Remediate code — replace deprecated calls with released APIs, CDS views, RAP.
🏗️
REWRITE
Criteria: Fundamentally incompatible — classic ALV reports, BDC-heavy programs, screen-based transactions.
Action: Rebuild using RAP + Fiori Elements + Clean Core patterns. High effort but future-proof.
✅
RETAIN
Criteria: Still needed, no deprecated elements, already Clean Core Level A or B. Unique business IP.
Action: Migrate unchanged. Validate on S/4HANA. Add to regression test suite.

Play 5 Quick Wins — RETIRE & REPLACE

RETIRE Checklist

  • Confirm with process owners that each program is truly unused

Step 5 — Decide Migration Approach
Greenfield (fresh install, re-implement) vs Brownfield (in-place conversion) vs Bluefield (selective data transition). This decision impacts every subsequent phase. If Fit-to-Standard and Clean Core are top priorities, Greenfield or Bluefield is recommended.

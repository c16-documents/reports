1 Executive Summary

🎯 Strategic Objective

Reduce the material creation cycle time in the Polaris MDG landscape from the current ~90 days (3 months) to a target of < 14 days (2 weeks), representing a minimum 60% reduction, while maintaining 100% GxP compliance and preserving the business requirement for partial-data material availability.

The current sequential Change Request (CR) chain — spanning FlowG initiation through GMID basic data creation, Polaris MDG data enrichment across 6 CR types, and IDOC distribution to downstream S/4HANA systems — creates extended lead times, multiple user dependencies, workflow queue bottlenecks, and delayed material availability for procurement and sales processes. This program proposes a combined strategy of parallel CR execution and intelligent automation to collapse the critical path while preserving every regulatory control and audit trail requirement.

2.2 Current CR Chain — Sequential Dependencies

The heart of the bottleneck lies in the sequential nature of six Change Request types within Polaris MDG. Each CR must be created, processed by a specific user persona, approved, and activated before the next can begin. The table below maps each CR type, its dependencies, and the nature of user interaction required.

📌 Key Observation — Stages 3a/3b/3c The plant and sales extension CRs (ZMAT02GP, ZMAT02GPS, ZMAT02GE) are inherently parallel — they address different data domains for different site types. However, all three currently depend on ZMAT02GB completion, creating an artificial serialization point. Furthermore, multiple instances of each type can be created for multi-site materials, compounding the wait time.

2.3 The ZMD_T_GMID_CR Configuration Table

The ZMD_T_GMID_CR table is the brain of CR auto-creation — it maps material type, plant, sales org, and other attributes to the appropriate CR types that should be triggered. When this table has gaps or missing mappings, the system cannot auto-create the required CRs, forcing manual intervention by an MDG administrator. These manual interventions are a hidden source of delay that doesn't appear in normal workflow metrics.

Configuration ElementPurposeFailure Impact
Material Type → CR Type Mapping Determines which CR types are triggered for a given material type Missing mapping → no auto-creation → manual admin intervention → 2-5 day delay
Plant → Site Category Mapping Identifies if plant is Athena-integrated, market/pivot, or S/4HANA production Wrong categorization → wrong CR type created → rework cycle
Sales Org Relevance Flags Determines if ZMAT02GPS is needed for sales-relevant plants Missing flag → ZMAT02GPS not created → sales data gap downstream
GxP Flag Determines if ZMAT02GX is required post-ZMAT02GE Missing flag → compliance step skipped → regulatory risk

3.3 Multi-Site Complexity Multiplier

For materials assigned to multiple sites, the system creates one CR instance per site for each relevant extension type. A material assigned to 5 Athena plants, 3 market sites, and 2 S/4HANA production plants would generate: 5× ZMAT02GP + 3× ZMAT02GPS + 2× ZMAT02GE + 2× ZMAT02GX = 12 parallel CRs in Stage 3+. Each requires individual user processing, creating a fan-out that dramatically increases the total human effort per material.

4.3 Dependency Analysis — What Really Needs to Be Sequential?

A critical question for simplification: which dependencies in the CR chain are technically mandatory (data integrity requires sequencing) versus historically assumed (designed this way but not technically required)? The following analysis reveals that only two of the four inter-CR dependencies are truly mandatory.

DependencyCurrentTechnically Mandatory?RationaleSimplification Opportunity
ZMAT01G → ZMAT02GB Sequential YES ZMAT02GB enriches the basic material record created by ZMAT01G. Without the base record, there is nothing to enrich. None — must remain sequential
ZMAT02GB → ZMAT02GP/GPS/GE Sequential NO Plant/sales extensions need the base material record (from ZMAT01G) but do not depend on the enriched basic data fields added by ZMAT02GB. Each extension operates on a separate set of views (plant data, sales data, industrial data). ✅ Can be parallelized with ZMAT02GB
ZMAT02GE → ZMAT02GX Sequential YES GxP approval (ZMAT02GX) requires the complete industrial view data from ZMAT02GE. The digital signature certifies the full data set — it cannot precede data entry. None — must remain sequential (GxP boundary)
ZMAT02GP/GPS → downstream Sequential YES IDOC distribution must happen post-activation to ensure data integrity in downstream systems. None — must remain sequential
✅ Critical Finding: The ZMAT02GB → Plant/Sales CR dependency is removable This single finding unlocks the most significant cycle time reduction. By triggering ZMAT02GP, ZMAT02GPS, and ZMAT02GE immediately after ZMAT01G activation (in parallel with ZMAT02GB), one full sequential stage is removed from the critical path. For the GxP pathway, this reduces the chain from 4 sequential stages to 3 — saving an estimated 20-25 days.

6.2 Modified CR Trigger Logic

The core technical change is modifying the event handler that fires after ZMAT01G activation. Today, this handler creates only ZMAT02GB and waits for its activation before creating plant/sales CRs. The modified logic creates all applicable CRs simultaneously.

EventCurrent BehaviorProposed BehaviorCode Change Location
ZMAT01G Activated Creates ZMAT02GB only Creates ZMAT02GB + ZMAT02GP + ZMAT02GPS + ZMAT02GE (where applicable per ZMD_T_GMID_CR) CR creation event handler class
ZMAT02GB Activated Creates ZMAT02GP/GPS/GE No action — plant/sales CRs already created at ZMAT01G stage Remove/bypass post-02GB trigger
ZMAT02GE Activated Creates ZMAT02GX Unchanged — ZMAT02GX still requires complete industrial data No change

6.3 Data Readiness Validation (Safeguard)

When plant/sales CRs run in parallel with ZMAT02GB, there is a scenario where the enriched basic data (from 02GB) might be needed by a plant CR. To handle this safely, a validation check is added to the plant/sales CRs that verifies base data completeness before allowing activation — not before allowing data entry. This means stewards can begin working on the CR immediately, but activation is gated on base data readiness.

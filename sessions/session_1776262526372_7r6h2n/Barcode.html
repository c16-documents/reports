Executive Summary

This document outlines a comprehensive warehouse management solution for Cadila Pharma, transitioning from legacy SAP transaction ZWM_N to a modern RAP-based Fiori Elements application. The solution implements 5 critical warehouse operations: Material Status Scan, Allocation to Rack, Rack-to-Rack Transfer, Issue to Dispense Area, and Return from Dispense Area.

The application leverages S/4HANA's RAP (RESTful ABAP Programming) framework with Fiori Elements V4 to provide a clean, intuitive user interface while maintaining full audit trail capabilities and integration with standard SAP material documents (MIGO).

Key Features and Functionality

KF-001: Material Status Scan

Purpose: Enable warehouse operators to scan material barcodes and retrieve comprehensive material information including current status, batch details, and location information.

Implementation Method: The feature will be implemented as a RAP action that accepts a material barcode input. The system will:

  • Parse the barcode to extract material number and batch
  • Query ZWM_BIN_STOCK to find current bin locations and quantities
  • Retrieve material master data from MARA table
  • Check material status by querying QM inspection lots (QALS) to determine Approved/Under Test/Rejected status
  • Fetch latest MIGO document details from MSEG/MKPF for GR quantity reference
  • Display consolidated information in a structured format

Floorplan Implementation: Implemented as an action button on the List Report. Users click "Material Status Scan", enter/scan the barcode, and receive a popup or side panel displaying the material information.

KF-002: Allocation to Rack

Purpose: Facilitate the allocation of materials from goods receipt to specific rack/bin locations while ensuring quantity validation against original GR documents.

Implementation Method: This feature will be implemented as a RAP create action that:

  • Accepts rack/bin barcode and material barcode inputs
  • Validates the destination bin is not marked as full (BIN_FULL_FLAG ≠ 'X')
  • Retrieves original GR quantity from MSEG/MKPF based on material and batch
  • Validates that allocation quantity does not exceed available GR quantity
  • Creates or updates record in ZWM_BIN_STOCK with new quantity
  • Logs transaction in ZWM_MOVEMENT_LOG with PROCESS_TYPE = 'ALLOC'
  • Updates audit fields with user and timestamp information

Floorplan Implementation: Accessible via "Allocate to Rack" action button. Opens a dialog with barcode input fields and quantity entry. Provides real-time validation feedback.

KF-003: Rack-to-Rack Transfer

Purpose: Enable transfer of materials between different rack/bin locations with support for partial quantities and full bin transfers.

Implementation Method: Implemented as a RAP action that:

  • Accepts source rack barcode and displays all materials in that location
  • Provides editable quantity fields for each material
  • Offers "Select All" functionality for complete bin transfer
  • Validates destination bin is not marked as full
  • Updates source bin quantities (subtract) in ZWM_BIN_STOCK
  • Updates destination bin quantities (add) in ZWM_BIN_STOCK
  • Automatically removes BIN_FULL_FLAG from source if quantity decreases
  • Logs transfer transaction in ZWM_MOVEMENT_LOG with PROCESS_TYPE = 'TRANSFER'
  • Deletes source records where quantity becomes zero

Floorplan Implementation: Multi-step dialog process: scan source rack → select materials/quantities → scan destination rack → confirm transfer.

KF-004: Issue to Dispense Area

Purpose: Manage the issuance of approved materials from rack locations to the dispense area with multi-rack picking capabilities.

Implementation Method: This feature will:

  • Accept material code and batch as input parameters
  • Validate material approval status through QM inspection lot checking
  • Query ZWM_BIN_STOCK to display all rack locations containing the material
  • Provide editable quantity fields with default values set to available quantities
  • Support line-item level confirmation for partial picking across multiple racks
  • Allow multiple materials to be added to the same transaction
  • Accept dispense area location barcode for final assignment
  • Update source rack quantities and create dispense area records
  • Log transaction with PROCESS_TYPE = 'ISSUE'

Floorplan Implementation: Complex dialog with material selection, multi-rack picking interface, and final dispense area assignment step.

KF-005: Return from Dispense Area

Purpose: Handle the return of materials from dispense area back to their original rack locations with validation and warning mechanisms.

Implementation Method: The return process will:

  • Accept material barcode from dispense area
  • Query ZWM_MOVEMENT_LOG to identify original rack/bin location
  • Check if original bin is marked as full (BIN_FULL_FLAG = 'X')
  • Display warning if returning to different bin than original
  • Allow user to select alternative bin if original is full
  • Update ZWM_BIN_STOCK to add returned quantity
  • Remove material from dispense area tracking
  • Log return transaction with PROCESS_TYPE = 'RETURN'
  • Provide confirmation of successful return

Floorplan Implementation: Simple dialog with barcode input, automatic original location lookup, and optional alternative location selection.

Business Rules

BR-001: Bin Full Status Management

Business Rule Description: Bin full status must be properly managed to prevent over-allocation and ensure accurate inventory tracking.

Key Features Addressed: KF-002 (Allocation to Rack), KF-003 (Rack-to-Rack Transfer), KF-005 (Return from Dispense Area)

Rule: When a bin is marked as full (BIN_FULL_FLAG = 'X'), no new allocations or transfers into that bin are permitted. However, movements out of a full bin are allowed and automatically remove the full status.

Implementation: Before any allocation or transfer operation, the system must check the destination bin's BIN_FULL_FLAG field in ZWM_BIN_STOCK. If the flag is set to 'X', the operation must be rejected with an appropriate error message. When any quantity is moved out of a bin, the system must automatically set BIN_FULL_FLAG to blank.

BR-002: GR Quantity Validation

Business Rule Description: Total allocated quantities cannot exceed the original goods receipt quantity for any material/batch combination.

Key Features Addressed: KF-002 (Allocation to Rack)

Rule: The system must validate that the sum of all allocated quantities for a material/batch combination does not exceed the original GR quantity as recorded in MSEG/MKPF.

Implementation: During allocation operations, query MSEG table to retrieve the latest material document quantity for the specific material and batch. Sum all existing MOVED_QTY values from ZWM_MOVEMENT_LOG where PROCESS_TYPE = 'ALLOC' for the same material/batch. Ensure new allocation plus existing allocations do not exceed GR quantity.

BR-003: Material Status Validation for Issue

Business Rule Description: Only approved materials can be issued to the dispense area; materials under test or rejected must be blocked from dispatch.

Key Features Addressed: KF-004 (Issue to Dispense Area)

Rule: Before allowing any issue operation, the system must verify that the material has an "Approved" status. Materials with "Under Test" or "Rejected" status must be prevented from being issued.

Implementation: Query QALS (QM Inspection Lot) table to determine the current inspection status for the material/batch combination. Check the STAT field for approved status codes. If status indicates under test or rejected, block the issue operation and display appropriate warning message.

BR-004: Audit Trail Completeness

Business Rule Description: All warehouse movements must be completely logged with full audit information for compliance and traceability.

Key Features Addressed: KF-001 through KF-005 (All operations)

Rule: Every warehouse operation must create a complete audit record including user, timestamp, quantities, source/destination locations, and process type.

Implementation: For every operation, create a record in ZWM_MOVEMENT_LOG with all required fields populated: MOVEMENT_ID (GUID), PROCESS_TYPE, material details, quantities, location information, and complete audit fields (CREATED_BY, CREATED_ON, etc.).

BR-005: Zero Quantity Record Cleanup

Business Rule Description: Stock records with zero quantities must be automatically removed to maintain data integrity.

Key Features Addressed: KF-003 (Rack-to-Rack Transfer), KF-004 (Issue to Dispense Area)

Rule: When a movement operation results in zero available quantity for a bin location, the corresponding record in ZWM_BIN_STOCK must be deleted.

Implementation: After updating AVAILABLE_QTY in ZWM_BIN_STOCK, check if the resulting quantity is zero. If so, delete the entire record to prevent accumulation of empty stock records.

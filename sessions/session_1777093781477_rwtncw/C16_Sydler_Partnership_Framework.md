## 1. Partnership Vision

> *"Sydler Technologies brings two decades of SAP domain expertise, trusted customer relationships, and proven service delivery. C16 brings an AI-native platform that reads, understands, analyses, and rewrites SAP custom code in minutes — work that traditionally takes consultants days or weeks.*
>
> *Together, we create something neither party can build alone: **The first AI-augmented SAP services practice that doesn't just find problems — it fixes them.**"*

The SAP ecosystem is at a generational inflection point. SAP's announcement of ECC end-of-mainstream-maintenance in 2027 (extended to 2030) has created an urgency that touches every SAP customer globally. These customers face three simultaneous challenges:

1. **Keeping ECC running reliably today** — operational excellence while senior developers retire or move on
2. **Preparing custom code for S/4HANA migration** — understanding 500–5,000 custom programs that nobody fully documents
3. **Retaining institutional knowledge** — the business rules encoded in custom ABAP live in the heads of a shrinking pool of experts

Traditional Application Management Services (AMS) address the first challenge — keeping ECC running. But the second and third challenges require a fundamentally different capability: the ability to understand custom code at scale, score it against SAP's evolving Clean Core standards, and produce actionable remediation — not in months, but in days. This is where C16 changes the equation.

**This partnership is not about replacing Sydler's consultants with AI.** It is about making every Sydler consultant dramatically more effective — enabling them to serve more customers, resolve incidents faster, and deliver insights that were previously impossible without weeks of manual analysis.

The result: **better service for customers, higher margins for Sydler, and a differentiated market position that no competitor can easily replicate.**

### Guiding Principle

> **Sydler is the trusted advisor. C16 is the intelligent engine.** The customer's relationship is with Sydler. The intelligence behind that relationship is powered by C16. Both brands are visible, but Sydler leads every customer interaction.

## 5. Technology Integration — How C16 Connects

### 5.1 Connection Architecture

C16 connects to the customer's SAP system via **standard SAP interfaces** — the same protocols used by SAP's own tools (Eclipse ADT, SAP GUI, OData services). No custom kernel modules, no database-level access, no operating system access. The connection is purely at the application layer.

| Component | Technology | Purpose |
|---|---|---|
| **RFC Connection** | SAP RFC (via SOAP over HTTPS) | Read table data, call function modules, read program source |
| **ADT Connection** | SAP ADT REST APIs (HTTPS) | Read/write ABAP objects, syntax check, ATC check, activation |
| **OData Connection** | SAP Gateway OData V2/V4 | Read business data for process intelligence (optional) |
| **Transport** | HTTPS (TLS 1.2+) | All communication encrypted in transit |

### 5.2 Graduated Access Model

C16 does **not** require production access in Phase 1. Custom code exists identically across DEV, QA, and Production systems. This eliminates the most common security team objection.

| Access Level | What It Enables | When Needed |
|---|---|---|
| **Level 1: Read-Only (DEV)** | Read source code, table structures, system configuration. No business data, no writes. | **Phase 1 — Proof of Value.** No security team friction. |
| **Level 2: Read + Deploy (DEV)** | Read + write ABAP objects to development system. No production access. | **Phase 2 — Active remediation.** Code deployed to dev/QA only. |
| **Level 3: Read Business Data (DEV/QA)** | Read transactional tables (EKKO, VBAK, BSIK) for process intelligence. | **Phase 3 — Process intelligence addon.** Optional. |

> ⚠️ **Security Principle:** C16 never requires production system access in Phase 1. Custom code analysis works identically on DEV, QA, or Sandbox — because the custom code is the same across all systems. This graduated access model eliminates the most common objection from customer IT security teams.

### 5.3 Data Handling & Compliance

| Concern | How C16 Handles It |
|---|---|
| Data at rest | Analysis results stored encrypted (AES-256-GCM). Customer can request purge at any time. |
| Data in transit | All communication over HTTPS (TLS 1.2+). No unencrypted channels. |
| Business data exposure | Phase 1–2 requires ZERO business data. Code analysis reads only ABAP source code and DDIC metadata. |
| Credential management | SAP service user credentials encrypted at rest. Never stored in plaintext. Never transmitted to third parties. |
| Multi-tenant isolation | Each customer's analysis runs in an isolated context. No cross-customer data leakage. |
| Audit trail | All C16 actions are logged with timestamp, user, object accessed, and action type. |

### 5.4 Onboarding Timeline — First Value in 7 Days

| Day | Activity | Owner | Duration |
|---|---|---|---|
| Day 1 | Customer provisions SAP service user with read-only role | Customer Basis + Sydler | 1–2 hours |
| Day 1 | C16 tests connectivity (RFC + ADT) | C16 team | 15 minutes |
| Day 2 | C16 runs initial portfolio scan (all Z-programs discovered) | C16 (automated) | 30 minutes |
| Day 2–3 | C16 generates first deliverable: Portfolio Health Report | C16 (automated) | 2–4 hours |
| Day 3–5 | Sydler reviews report, adds business context, prepares customer presentation | Sydler functional team | 1 day |
| Day 5–7 | Joint presentation of findings to customer | Sydler (lead) + C16 (technical backup) | 1 hour |

> ✅ **From the moment a customer says "yes," they see actionable intelligence from their own SAP system within 7 calendar days.** This is the single most powerful differentiator versus any competitor.

## 12. Brand Guidelines

### 12.1 Brand Positioning

Both brands are visible in all joint activities. The positioning follows a clear principle:

> **Sydler is the service brand (front stage). C16 is the technology brand (powering the service).**

| Touchpoint | Brand Treatment |
|---|---|
| **Customer proposals** | Sydler-branded proposal with "Powered by C16 AI Platform" designation. C16 logo in technology section, not cover page. |
| **Analysis reports** | C16-generated reports carry C16 header. "Delivered by Sydler Technologies" in footer. Customer sees both brands. |
| **Customer meetings** | Sydler leads all meetings. C16 joins for technical deep-dives or demos when invited by Sydler. |
| **Support interactions** | All communication through Sydler's support desk. Customer never contacts C16 directly. |
| **Marketing materials** | Joint case studies, blog posts, event presentations. Both logos. Sydler quoted on service, C16 on technology. |
| **Contracts** | Customer's contract is with Sydler. Sydler's contract with C16 is a separate partnership agreement. |

### 12.2 Joint Identity

> **Suggested Tagline:**
> 
> ### *"Intelligent SAP Services — Expert Delivery, AI-Powered Insight"*
> 
> *Sydler Technologies × C16*

### 12.3 Do's and Don'ts

| ✅ Do | ❌ Don't |
|---|---|
| Refer to "our AI platform" or "C16, our technology partner" | Position C16 as a competitor or alternative to Sydler |
| Show C16 reports as part of service delivery — they demonstrate capability | Let customers engage C16 directly for services (bypass Sydler) |
| Credit both parties in case studies and references | Claim C16 capabilities as Sydler-built, or vice versa |
| Allow C16 branding on technical reports — builds credibility | Remove C16 branding — creates confusion and breaks trust if discovered |
| Position as "best of both worlds" — AI speed + human wisdom | Position AI as replacing human consultants — creates internal resistance |

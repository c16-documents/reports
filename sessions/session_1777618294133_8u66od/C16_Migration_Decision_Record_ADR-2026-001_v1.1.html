1 Decision Trigger

In April 2026, SAP issued a refreshed SAP API Policy that formalises a hard contractual line between Published APIs and everything else. Four specific provisions create compliance exposure for C16:

ClauseVerbatim Excerpt (from SAP API Policy PDF)C16 Exposure
§1.1 — Published APIs "APIs published on the SAP Business Accelerator Hub… or otherwise identified in product-specific Documentation ('Published APIs') are made available for the purposes described in that Documentation." C16 uses ADT APIs which are not listed on the API Hub as Published APIs for customer/third-party consumption.
§1.2 — Non-Published APIs "Customer and third-party applications must not access, invoke, or interact in any manner with APIs that are not Published APIs… except as permitted by the Documentation or otherwise authorized by SAP (e.g., customers may use custom-developed ABAP interfaces in private cloud and on-premise deployments)." The named exception for custom-developed ABAP interfaces is C16's primary compliance path. C16's ICF endpoints (/sap/bc/zc16/*) qualify under this exception.
§3 — Anti-Circumvention "Customers, partners, and third parties must not bypass, disable, or otherwise circumvent API Controls, including through intermediary services, custom code or developments, proxies, gateways, impersonation techniques, or similar mechanisms." C16 SaaS (Azure) currently acts as an intermediary between the user and SAP. The split-plane Connector architecture addresses this.
api.sap.com ToU §3(m) "use any automated data gathering or extraction methods… (such as bots, scrapers, spiders, crawlers, Model Context Protocol (MCP), agent-frameworks, or any other text and data mining technology)… for artificial intelligence training, improvement or evaluation" C16's sap_api_hub_search tool makes automated calls to api.sap.com. This is a separate legal instrument from the API Policy but creates additional exposure.

⚠️ Important Interpretive Note

The official SAP API Policy PDF does not contain explicit, named anti-AI-agent language. References to "Section 2.2.2" in third-party reporting could not be verified against the published text. The AI implications are interpretive consequences of §1.2 (non-published API prohibition), §2 (bulk extraction controls), and §3 (anti-circumvention) read together, reinforced by SAP leadership statements distinguishing customer data from SAP's "domain knowhow / IP."

Source: SAP API Policy PDF verified 2026-05-01 at https://help.sap.com/doc/sap-api-policy/latest/en-US/API_Policy_latest.pdf

Key Problems in As-Is

  1. AI directly touches SAP — the LLM (Claude) plans and executes API call sequences against the customer's SAP system. This matches the interpreted 2.2.2(a) pattern.
  2. 29 tools use ADT APIs — ADT is not formally listed on api.sap.com as a Published API for customer/third-party consumption.
  3. SAP credentials in Azure Key Vault — credentials leave the customer's environment.
  4. No SAP-endorsed architecture in path — VPN is network infrastructure, not an SAP product.
  5. api.sap.com dependency — sap_api_hub_search makes automated calls to SAP's website.
  6. Source code extraction — full ABAP source transferred to C16 cloud for analysis.

5 Decision

✅ DECIDED: Option A (Bridge) + Option B (Destination)

Option A is the bridge. Complete the ICF migration, remove api.sap.com, add human approval — these are immediate risk-reduction actions that can ship in Q2-Q3 2026 with minimal architecture change.

Option B is the destination. Build the C16 Connector as the strategic target architecture. AI never touches SAP. Credentials stay in customer environment. Analysis runs in-system. This is the structural fix.

Option D is an optional enhancement. For customers with BTP, add Cloud Connector on top of Option B for maximum compliance. Not required but recommended for RISE customers and regulated industries.

Option C is reserved for regulated industries (banking, defense, government) and air-gapped environments where no data can leave the customer network under any circumstances.

Key Architectural Principles

  1. AI never touches SAP. The LLM (Claude) plans analysis workflows and generates signed instructions. It never executes an API call against any SAP system.
  2. The Connector is NOT an AI system. It is a deterministic execution engine — same architectural pattern as SAP Cloud Connector, GitHub self-hosted runners, Azure Arc agents, Datadog agents.
  3. Customer owns everything SAP-side. ICF endpoints are Z-namespace, customer-transported, customer-authorized. Credentials never leave customer environment.
  4. Released ABAP only. ICF endpoints wrap only cloudification-verified released SAP classes. Verified quarterly.
  5. Full audit trail. Every SAP call logged in ZC16_API_LOG (customer-side) and Connector local audit log.
  6. Source code managed via GitHub. C16 platform, Connector images, ICF endpoint ABAP, and skills all version-controlled. Customer's generated code optionally version-controlled via abapGit.

9.2 — abapGit: Role in the Architecture

ℹ️ What is abapGit?

abapGit is an open-source Git client for ABAP, hosted on GitHub. It is the de facto standard for ABAP version control, with active contributions from SAP employees. SAP recognizes abapGit as a community tool and promotes its use for custom ABAP development.

abapGit serves two roles in C16's architecture:

RoleHow It's UsedPolicy Assessment
1. Deployment Mechanism C16's sync_abap_git_objects tool calls the customer-installed ZCL_C16_ABAPGIT class, which uses abapGit's serialization/deserialization classes to create, update, and activate ABAP objects. This is the same mechanism used by abapGit's own UI and by SAP's gCTS. LOW RISK — abapGit runs INSIDE the customer's SAP system. It is customer-installed, customer-owned code. SAP actively promotes abapGit. The underlying classes are well-established in the SAP ecosystem.
2. Customer Code Version Control (Future) In the Connector architecture, customer-generated ABAP code (reports, classes, CDS views) can optionally be version-controlled in the customer's own Git repository via abapGit. This creates a full audit trail of every code change C16 makes. LOW RISK — Customer's own repo, customer's own review process. Strengthens the "customer-developed" argument. abapGit does not interact with api.sap.com or any SAP cloud service.

✅ abapGit Compliance Position

abapGit is an argument in C16's favor, not a risk. If SAP challenged C16's use of abapGit, they would be challenging their own community ecosystem. abapGit is:

  • Open-source with SAP employee contributors
  • Installed on the customer's own system as customer code
  • Used by thousands of SAP customers and partners globally
  • The foundation of SAP's own gCTS (Git-enabled Change & Transport System)

✅ Why This Strengthens §1.2 Compliance

By providing ABAP source code to the customer (not compiled binaries), the customer:

  • Reviews every line before installing — they know exactly what runs on their system
  • Compiles the code in their own system — it becomes their binary
  • Transports through their own CTS — it follows their change management
  • Owns the installed objects — Z-namespace, customer package, customer transport
  • Can modify if needed — they have the source, not a black box

This makes the "customer-developed ABAP interface" characterization unambiguous. The customer reviewed it, compiled it, transported it, and operates it. It is their code.

For maximum compliance, customer-generated code should be in the customer's own Git repository, not C16's. This ensures the customer maintains full ownership of all code artifacts, with an independent audit trail that is not controlled by C16.
